Network Anomaly
Detector

An unsupervised machine learning system that detects unusual KPI patterns in XLSMART's network data — catching degradations before they escalate into alarms and giving the operations team an early warning system for network health issues.

Python Unsupervised ML Apache Spark SmartCare Anomaly Detection Scikit-learn Telecom
// The Problem

XLSMART's current monitoring relies on threshold-based alarms — an alarm fires only after a KPI has already breached a hard limit. This means the team is always reacting after the fact. Subtle degradation patterns — gradual drops, unusual cluster behavior, or multi-KPI correlations — go undetected until they become critical failures.

// The Solution

By applying unsupervised anomaly detection on SmartCare KPI streams via Spark, the system learns what "normal" looks like for each network element and flags deviations before thresholds are breached. No labeled data needed — the model detects anomalies purely from patterns in the data itself.

// candidate algorithms
Algorithm Candidates
Isolation Forest
Strong candidate
Isolates anomalies by randomly partitioning data. Points that are isolated quickly (short path) are flagged as anomalies. Works well on high-dimensional telecom KPI data without assumptions about distribution.
✓ Fast · scalable · no distribution assumption
DBSCAN
Strong candidate
Density-based clustering that marks low-density points as outliers. Ideal for detecting spatially isolated KPI behavior across site clusters — no need to pre-define number of clusters.
✓ No k needed · handles noise naturally
K-Means Clustering
Candidate
Groups KPI behavior into clusters. Points far from any cluster centroid are treated as anomalies. Simple, interpretable, and fast — useful as a baseline comparison model.
✓ Interpretable · good baseline
Local Outlier Factor
Candidate
Compares local density of each point to its neighbors. Effective at detecting anomalies in areas where "normal" KPI ranges vary between regions or time periods.
✓ Context-aware · good for varying baselines
// features
Planned Features
Pre-Alarm Detection
Flags anomalies before KPIs breach hard thresholds — giving engineers a window to investigate and act before users are impacted.
Per-Node Scoring
Each site, MME, and PGW gets an anomaly score per KPI — enabling targeted investigation rather than blanket network-wide alerts.
Scheduled Detection Runs
Model runs on a defined schedule against fresh SmartCare data — continuously updating anomaly scores as network conditions evolve.
Multi-Model Comparison
Run Isolation Forest, DBSCAN, and LOF in parallel — compare results to select the best-performing algorithm for XLSMART's specific KPI patterns.
Auto Demarcation Integration
Anomaly flags can feed directly into the Auto Demarcation pipeline — enriching root cause analysis with pre-alarm context for faster diagnosis.
Tunable Sensitivity
Contamination rate and threshold parameters are configurable per KPI type — balancing false positive rate against detection sensitivity for each use case.
